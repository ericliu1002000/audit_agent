# 审计智能体项目开发规范（草案）

本文档用于约束和指导本项目的日常开发，保证代码结构清晰、可维护、易协作。

## 1. 环境与依赖管理

- 统一使用 Conda 环境：开发前先执行  
  `conda activate audit_agent`
- 所有第三方依赖必须写在 `requirements.txt` 中，禁止在本地临时安装但不写入依赖文件。
- 依赖升级或新增时，说明原因并在 PR/提交说明中简单标注。

## 2. 后端分层与目录约定

- 一个数据库表对应一个 `model.py` 文件，保持模型文件职责单一、结构清晰。
- 模型相关代码分为三层：
  - **Model**：字段定义、基础属性、简单方法；
  - **Manager**：查询封装、复杂 Query 逻辑；
  - **Service**：业务逻辑聚合层，**所有业务逻辑尽量放在 Service 中**，避免散落在 View 或 Model 里。
- View 统一采用 **类视图（Class-Based View）**，保持风格一致、便于复用和扩展。
- 编写代码时优先考虑可读性和可维护性，避免过度“技巧化”的写法。

## 3. 代码风格与配置

- 代码规范统一通过 `pyproject.toml` 进行配置（如 Black、isort、ruff/flake8 等），避免多人维护多个配置文件。
- 提交代码前应至少保证：无语法错误、基本风格检查通过。
- 命名建议：见名知意，避免使用单字母或难以理解的缩写。

## 4. 测试策略

- 开发阶段可以不立刻写测试，但**功能开发完成并确认结束后，需要集中补齐测试用例**。
- 测试目标：功能点覆盖率 **≥ 80%**，重点关注：
  - 核心业务流程；
  - 边界条件与异常分支；
  - 权限、审计、风控等关键逻辑。
- 测试脚本与数据统一放在约定的 `tests/` 目录中（后续可细化结构，如按模块拆分子目录）。

## 5. 视图与业务代码规范

- View 负责：请求解析、参数校验、权限检查、调用 Service、返回统一格式的响应。
- Service 负责：业务流程编排、调用 Manager 和其他 Service、统一处理业务异常。
- Model/Manager 尽量保持“无业务”的数据层职责，避免和具体业务强耦合。
- 所有面向外部的接口尽量保持风格一致（URL 命名、HTTP 方法、返回结构）。

## 6. 设计与前端相关（补充）

- 颜色、组件等视觉规范统一记录在 `docs/design/` 下，例如 `docs/design/color-palette.md` 中的“现代智能 - 科技蓝”配色方案。
- 前端、后端在使用颜色与状态时优先使用已定义的 Design Token（如 CSS 变量），避免随意新增自定义颜色。
